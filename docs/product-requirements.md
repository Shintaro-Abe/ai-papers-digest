# プロダクト要求定義書（PRD）

## 1. プロダクトビジョンと目的

### ビジョン
AI分野の最新論文を自動的にキュレーション・要約し、AIエンジニアが効率的に技術キャッチアップできるインテリジェントなエージェントシステム **「AI Papers Digest」** を構築する。

### 目的
- AI関連の論文が日々大量に公開される中で、質の高い論文のみを自動抽出する
- AIエンジニア向けに構造化された日本語要約を自動生成し、短時間で論文の本質を把握可能にする
- ユーザーフィードバックを蓄積し、要約品質と論文選定精度を継続的に改善する

### 既存ツールとの差別化

| ツール | 特徴 | 本プロダクトとの違い |
|--------|------|---------------------|
| arXiv | 論文の一次ソース。フィルタリングなし | 本プロダクトは注目度ベースの自動フィルタリングを提供 |
| Hugging Face Papers | コミュニティによるキュレーション + upvote | 本プロダクトはさらに要約生成・パーソナライズを追加 |
| Semantic Scholar | 引用数・TLDR・推薦機能あり | 本プロダクトはAIエンジニア特化の構造化要約と日本語対応 |
| Papers with Code | コード付き論文に特化 | 本プロダクトは実装可能性評価を含む包括的な要約を提供 |
| Elicit | AI論文検索・要約ツール | 本プロダクトは日次プッシュ型・フィードバック学習付き |

**本プロダクトのユニークバリュー:**
1. **日次プッシュ型** — ユーザーが検索しなくても毎朝届く
2. **AIエンジニア特化の構造化要約** — 新規性・手法・実装可能性を明示
3. **フィードバック学習** — 個人の興味に適応していく
4. **日本語対応** — 英語論文の障壁を下げる

## 2. ターゲットユーザーと課題・ニーズ

### ターゲットユーザー
- 社内のAIエンジニア（初期ユーザー: 数名〜数十名規模）
- 高難度のAI関連タスクに取り組み、最新論文の実装が必要な技術者

### ペルソナ
> **田中さん（30歳・AIエンジニア歴3年）**
> - 業務で LLM ベースのシステムを開発中
> - 最新の Attention 機構や学習手法を論文から学んで実装に活かしたい
> - 英語論文は読めるが、毎日10本以上スクリーニングする時間はない
> - 朝の15分で「今週読むべき論文」を把握したい

### 課題
- arXivに日々投稿されるAI関連論文は膨大（cs.AI + cs.CL + cs.CV + cs.LG で 1日100本以上）
- どの論文が注目されているか・質が高いかの判断に時間がかかる
- 英語論文のアブストラクトを読むだけでも1本5分 × 数十本は非現実的
- 既存ツールは検索（プル型）が主で、日次のプッシュ型配信がない

### ニーズ
- 注目度の高い論文を自動的にフィルタリングしてほしい（1日5〜10本に絞る）
- AIエンジニアとして「何が新しいか」「実装に使えるか」がわかる構造化要約がほしい
- 日本語で簡潔に読みたい
- 自分の興味・評価に基づいて、今後のレコメンド精度を高めたい

### LLM コスト戦略: Claude Max プラン活用

コストを抑えるため、Claude Max プラン（サブスクリプション）を最大限活用する。

> **Claude Max プランの技術仕様:**
> - Anthropic API（HTTPエンドポイント）/ Agent SDK / Bedrock では使用不可
> - プログラムから利用する方法: **`claude -p`（CLI パイプモード）**
>   - `claude -p "プロンプト"` で非対話的に実行し、結果を標準出力に返す
>   - `--output-format json` で JSON パース可能な出力を取得
>   - stdin パイプ対応: `echo "論文データ" | claude -p "要約して"`
>   - Max プランの課金内で実行されるため **追加API費用なし**
>
> **アーキテクチャへの影響:**
> - Lambda から直接 Claude API を呼ぶのではなく、**`claude -p` を実行可能な環境**が必要
> - 選択肢:
>   - **EC2 / ECS**: Claude CLI をインストールした環境で `claude -p` を実行
>   - **CodeBuild**: ビルド環境として Claude CLI を実行（日次バッチ向き）
>   - **Lambda + EFS**: CLI バイナリを EFS に配置して Lambda から実行（制約あり）
> - **推奨**: **ECS Fargate タスク**（日次起動、処理完了で停止、コスト最小）
>   または **CodeBuild**（ビルドプロジェクトとしてバッチ実行）
>
> **フォールバック:**
> - Max プランのレート制限に達した場合のみ、Anthropic API（従量課金）をフォールバックとして使用

## 6. ユーザーストーリー

### US1: 毎朝の論文要約を Slack で確認する
> AIエンジニアとして、毎朝 Slack チャンネルに届く注目AI論文の日本語要約を確認したい。
> それにより、朝の15分で今日知るべきAI動向を把握できる。

**受け入れ条件:**
- 毎朝 JST 8:00 までに Slack に配信される
- 1日5〜10本の論文が選定される
- 各要約に arXiv リンクが含まれる

### US2: Slack でサッと概要を掴み、詳細ページで深堀りする
> AIエンジニアとして、Slack では200〜400文字のコンパクト要約で概要を掴み、気になる論文は詳細ページで深堀りしたい。
> それにより、Slack のスクロール負荷なく効率的にスクリーニングできる。

**受け入れ条件:**
- Slack メッセージ（コンパクト要約）:
  - 📄 タイトル（原題 + 日本語訳）
  - 🎯 要約本文（200〜400文字。新規性・手法・結果のエッセンスを凝縮）
  - 🏷️ タグ（分野タグ: LLM, Vision, RL 等）
  - 🔗 リンク（「詳細を見る」→ S3詳細ページ / arXiv）
- 詳細ページ（S3静的サイト）:
  - 🆕 新規性（何が新しいか）
  - 🔧 手法（どうやっているか）
  - 📊 結果（どんな成果か）
  - 💡 実装可能性（実務で使えるか）
  - 論文メタデータ（著者、カテゴリ、upvote数、引用数等）

### US3: 要約を評価する
> AIエンジニアとして、要約に対して Slack リアクション（👍/👎）で評価したい。
> それにより、自分にとって有用な論文が今後より多く表示されるようになる。

**受け入れ条件:**
- Slack メッセージに 👍/👎 リアクションで評価できる
- 評価結果がデータベースに保存される
- 評価データが次回以降のスコアリングに反映される

### US4: 過去の要約を検索する
> AIエンジニアとして、過去に配信された要約をキーワードや分野で検索したい。
> それにより、必要な時に特定の論文情報にアクセスできる。

**受け入れ条件:**
- Web ダッシュボードでキーワード検索ができる
- 分野タグでフィルタリングできる
- 類似論文のレコメンドが表示される

## 7. データソース仕様（検証済み）

### arXiv API
- **エンドポイント**: `https://export.arxiv.org/api/query`
- **対象カテゴリ**: `cs.AI`, `cs.CL`, `cs.CV`, `cs.LG`, `stat.ML`
- **レート制限**: 3秒に1リクエスト
- **レスポンス形式**: Atom XML
- **取得データ**: タイトル、著者、アブストラクト、カテゴリ、公開日、arXiv ID

### Hugging Face Papers API（認証不要）
- **Daily Papers**: `GET https://huggingface.co/api/daily_papers?date=YYYY-MM-DD`
  - 返却データ: arXiv ID、タイトル、要約、upvote数、ai_summary、ai_keywords、githubRepo
  - **日付はUTC基準**（JSTではない）。パイプライン実行時にUTC当日＋UTC前日の2日分を取得し、arXiv IDで重複排除
  - 明示的な日付指定（バックフィル/リトライ）時は、指定日（JST）からUTC日付を導出
- **個別論文**: `GET https://huggingface.co/api/papers/{arxiv_id}`
- **論文検索**: `GET https://huggingface.co/api/papers?query={query}&limit={n}`
- **Trending**: 専用APIなし → Daily Papers を upvote 数でソートして代替
- **レート制限**: 明示的な制限なし（読み取り専用）

### Semantic Scholar API
- **エンドポイント**: `https://api.semanticscholar.org/graph/v1/`
- **主要API**: `/paper/search`, `/paper/{id}`, `/paper/batch`（POST）
- **取得可能フィールド**: citationCount, tldr, embedding（SPECTER2）, referenceCount
- **レート制限**: 認証済みで 1 RPS（APIキー無料取得可能）
- **活用方法**: 引用数によるスコアリング補強、TLDR との品質比較

## 8. 機能要件

### FR1: データ収集エンジン
- **arXiv**: 日次で対象カテゴリの新着論文メタデータを取得（Search API）
- **Hugging Face**: Daily Papers API でUTC当日+前日の2日分の論文 + upvote数を取得（日付間の重複排除あり）
- **Semantic Scholar**: arXiv IDをキーに引用数・TLDRを補完取得（batch API）
- **重複排除**: arXiv ID をユニークキーとして統合（HFの日付間、HF+arXiv間の両方）
- **バックフィル**: 明示的な日付指定でリトライ・過去データ再収集が可能
- **スケジュール**: EventBridge による日次実行（JST 6:00）

### FR2: 論文スコアリング・フィルタリング
- **注目度スコア算出ロジック**:
  ```
  score = w1 × hf_upvotes（正規化）
        + w2 × semantic_scholar_citations（正規化）
        + w3 × source_count（何ソースに出現したか）
        + w4 × feedback_bonus（カテゴリベースのLike傾向に基づく加点）
  ```
  - 初期ウェイト: w1=0.4, w2=0.2, w3=0.2, w4=0.2（週次で動的調整）
  - feedback_bonus: 直近28日の👍/👎からカテゴリ別Like率を計算（Laplace平滑化）
- **フィルタリング**: スコア上位5〜10本を選出（閾値は設定可能）
- **重複排除**: 過去に配信済みの arXiv ID は除外

### FR3: 要約生成エンジン（2層構造）
- **LLM**: Claude Max プラン経由 `claude -p` CLI（Opus 4.6 を基本）
- **実行環境**: ECS Fargate タスク（Claude CLI インストール済みコンテナ）
- **入力**: 論文タイトル + arXiv アブストラクト + Semantic Scholar TLDR + HF ai_summary
- **出力（2層構造）**:
  - **コンパクト要約（Slack 配信用）**: 200〜400文字。新規性・手法・結果のエッセンスを1段落に凝縮
  - **詳細要約（S3 詳細ページ用）**: 構造化フォーマットで全セクションを記述
    - 🆕 新規性（何が新しいか、2-3文）
    - 🔧 手法（技術的アプローチ、2-3文）
    - 📊 結果（主要な実験結果・ベンチマーク、2-3文）
    - 💡 実装可能性（実務での活用可能性、1-2文）
- **品質比較**: HF ai_summary が存在する場合、自動生成要約との品質スコアを比較
  - 品質スコア = 構造化度 + 情報量 + 日本語の自然さ（LLM-as-judge で評価）
  - 高スコアの要約を採用（ただし2層構造テンプレートへの整形は実施）

### FR4: 詳細ページ生成・ホスティング
- **S3 静的サイト**: 論文ごとに HTML ページを生成し S3 + CloudFront でホスティング
- **ページ内容**: 詳細要約全文 + 論文メタデータ + arXiv/GitHub リンク
- **URL 形式**: `https://{domain}/papers/{arxiv_id}.html`
- **生成タイミング**: 要約生成後、Fargate タスク内で HTML を生成し S3 にアップロード

### FR5: Slack 配信システム
- **Slack 配信**: Slack Incoming Webhook で指定チャンネルに投稿
- **フォーマット**: Slack Block Kit でコンパクトなメッセージ形式
- **メッセージ内容**: タイトル + コンパクト要約（200〜400文字）+ タグ + リンクボタン
- **リンクボタン**: 「📋 詳細を見る」→ S3詳細ページ / 「📖 arXiv」→ arXiv
- **配信タイミング**: 日次 JST 8:00（収集→要約→ページ生成→配信のパイプライン完了後）

### FR6: フィードバックシステム（Phase 2）
- **収集方法**: Slack Events API で 👍/👎 リアクションを検知
- **データモデル**: `{user_id, paper_id, reaction, timestamp}`
- **学習ロジック**:
  - 日次: Like した論文の arXiv カテゴリ傾向から `feedback_bonus` を計算（Laplace 平滑化）
  - 週次: 各特徴量の予測力をベイズ平滑化で評価し、EMA ブレンドでウェイト w1-w4 を調整
  - 安全策: 最低フィードバック数（5件）未満はウェイト変更しない

### FR7: データストア
- **DynamoDB テーブル設計**:
  - `papers`: PK=arxiv_id / 論文メタデータ、スコア、ソース情報
  - `summaries`: PK=arxiv_id / 要約テキスト、品質スコア、生成日
  - `feedback`: PK=user_id, SK=arxiv_id / リアクション、タイムスタンプ
  - `delivery_log`: PK=date / 配信済み論文ID一覧、配信ステータス

### FR8: セマンティック検索（Phase 3）
- **ベクトルストア**: Amazon S3 Vectors（vector bucket + vector index）
  - プロビジョニング不要、ゼロスケール、従量課金
  - 月額 < $0.05（本アプリの規模）
- **埋め込み**: Bedrock Titan Embeddings（768次元）または Semantic Scholar SPECTER2 embedding
- **距離メトリクス**: cosine（コサイン類似度）
- **メタデータフィルタリング**: タグ、日付、カテゴリでクエリ結果をフィルタ
- **用途**: 過去要約の類似検索、関連論文レコメンド

## 9. 非機能要件

### パフォーマンス
- 日次バッチ処理（収集→要約生成→配信）は30分以内に完了
- Web ダッシュボードのレスポンスは2秒以内（Phase 3）
- Semantic Scholar API のレート制限（1 RPS）を考慮したバッチ設計

### 可用性
- 日次バッチ: 95%以上の成功率（Phase 1）、99%以上（Phase 2〜）
- Web ダッシュボード: 99%以上（Phase 3）
- バッチ失敗時のリトライ機構（最大3回、指数バックオフ）
- Dead Letter Queue（DLQ）によるエラーハンドリング

### セキュリティ
- AWS IAM による最小権限の原則
- API キー（Anthropic, Semantic Scholar, Slack）は AWS Secrets Manager で管理（ハードコードなし）
- 外部API通信はHTTPS必須
- Slack Webhook URL の秘匿管理
- DynamoDB の暗号化（保存時暗号化はデフォルト有効）+ 全テーブル PITR 有効
- CloudFront 地理的制限: 日本（JP）のみ許可（個人利用・日本限定アクセス）
- S3 パブリックアクセス全ブロック（CloudFront OAC 経由のみ）
- Fargate セキュリティグループ: インバウンド全拒否
- ECR: イメージタグ IMMUTABLE + プッシュ時脆弱性スキャン有効
- SQS DLQ: SSE-SQS によるキュー暗号化
- SNS: aws/sns マネージドキーによるトピック暗号化
- セキュリティスキャン: Trivy v0.69.3 で IaC 監査実施済み（詳細は architecture.md §12）

### スケーラビリティ
- サーバーレス構成（Lambda + DynamoDB）により自動スケール
- 論文数の増加に対してコスト線形増加
- Phase 3 の S3 Vectors はゼロスケール（使用量に応じた従量課金）

### 監視・運用
- CloudWatch Logs によるバッチ実行ログの集中管理
- CloudWatch Alarms: バッチ失敗、Lambda エラー率、DynamoDB スロットリング
- CloudWatch Metrics: 日次処理件数、要約生成時間、配信成功率
- Terraform による Infrastructure as Code（全リソース管理）

### コスト管理
- AWS Budgets による月次予算アラート（$50閾値）
- Lambda の同時実行数制限
- DynamoDB オンデマンドキャパシティ（予測不要）
